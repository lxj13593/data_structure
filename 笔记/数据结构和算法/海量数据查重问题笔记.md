# 海量数据查重问题笔记

## 题目一：内存足够时，查找两个文件中的重复数据

### 题目描述

有两个文件 `a` 和 `b`，文件中存放了大量 IP 地址、URL 地址或 Email 地址。

要求找出两个文件中重复出现的数据并输出。

---

## 一、基本思路

如果内存足够，可以把文件 `a` 中的所有数据存入哈希表，再依次读取文件 `b` 中的数据。

处理过程：

1. 读取文件 `a` 中的所有数据；
2. 将每条数据插入哈希表；
3. 依次读取文件 `b` 中的数据；
4. 每读取一条，就在哈希表中查找；
5. 如果查找到，说明该数据同时存在于两个文件中，将其输出。

例如：

```text
文件 a：
192.168.1.1
192.168.1.2
192.168.1.3

文件 b：
192.168.1.2
192.168.1.4
192.168.1.3
```

将文件 `a` 放入哈希表后，依次查询文件 `b`，最终找到：

```text
192.168.1.2
192.168.1.3
```

---

## 二、为什么使用哈希表

哈希表的单次插入和查找，平均时间复杂度都是：

```text
O(1)
```

所以：

- 文件 `a` 中有 `n` 条数据，建立哈希表需要 `O(n)`；
- 文件 `b` 中有 `m` 条数据，逐条查询需要 `O(m)`。

整个算法的平均时间复杂度是：

```text
O(n + m)
```

不是 `O(1)`。

`O(1)` 只表示单次哈希查询的平均复杂度。

---

## 三、空间复杂度

需要把文件 `a` 中的数据放入哈希表，因此空间复杂度大约是：

```text
O(n)
```

也可以选择将较小的文件放入内存，再扫描较大的文件，这样可以减少内存占用。

---

## 四、需要注意重复输出

如果一个数据在文件 `b` 中重复出现多次，直接查询并输出可能导致同一个数据被输出多次。

例如：

```text
文件 a：
1
2
3

文件 b：
2
2
2
```

可能输出：

```text
2
2
2
```

如果题目要求每个重复数据只输出一次，可以再使用一个集合记录已经输出过的数据。

---

# 题目二：内存限制为 100 MB 时，查找两个大文件中的重复 IP

## 题目描述

有两个文件 `a` 和 `b`，每个文件中大约有 1 亿条 IPv4 地址。

每个 IPv4 地址可以用一个 32 位整数表示，占 4 字节。

限制内存只能使用 100 MB，要求找出两个文件中重复的 IP 地址并输出。

---

## 一、为什么不能直接全部放入内存

一个文件中有 1 亿条 IP，每条占 4 字节：

```text
1 亿 × 4 字节 = 4 亿字节
```

大约是：

```text
400 MB
```

两个文件的原始数据总量约为：

```text
800 MB
```

已经超过 100 MB 的内存限制。

而且如果使用 `unordered_set`，每个元素除了 4 字节 IP 以外，还会有：

- 哈希桶；
- 结点；
- 指针；
- 对齐空间；
- 容器自身的管理开销。

所以实际内存占用会远大于 400 MB。

因此不能直接将整个文件放进哈希表。

---

## 二、核心思路：哈希分桶

将两个大文件分别拆分成多个小文件。

对每个 IP 计算：

```text
hash(IP) % bucket_count
```

得到桶下标。

例如分成 100 个桶：

```text
文件 a：
a_0、a_1、a_2、……、a_99

文件 b：
b_0、b_1、b_2、……、b_99
```

读取文件 `a` 中的每个 IP：

```text
hash(IP) % 100 = 7
```

就把它追加写入 `a_7`。

读取文件 `b` 时使用完全相同的哈希函数和桶数：

```text
hash(IP) % 100 = 7
```

就把它写入 `b_7`。

---

## 三、为什么相同 IP 一定进入对应的同一个桶

对于相同的 IP：

```text
IP 相同
→ 哈希值相同
→ 对相同桶数取余的结果相同
→ 桶下标相同
```

因此，如果某个 IP 同时存在于文件 `a` 和文件 `b` 中，它一定会进入同编号的一对桶：

```text
a_0 和 b_0
a_1 和 b_1
a_2 和 b_2
……
```

所以不需要比较：

```text
a_1 和 b_2
a_3 和 b_8
```

只需要比较编号相同的桶。

---

## 四、逐桶查找重复数据

分桶完成后，一次只处理一对小文件。

例如比较：

```text
a_0 和 b_0
```

处理方式：

1. 将 `a_0` 中的数据读入哈希表；
2. 流式读取 `b_0`；
3. 每次从 `b_0` 中只读取一条 IP；
4. 在哈希表中查找；
5. 查找到则输出；
6. 处理完成后释放哈希表；
7. 再处理 `a_1` 和 `b_1`。

这里不需要把 `a_0` 和 `b_0` 同时全部放入内存。

内存中主要保存的是：

```text
a_0 对应的哈希表
+
b_0 当前读取的一条数据
```

也可以把更小的那个桶放入哈希表，然后扫描另一个桶。

---

## 五、图中 80 MB 的含义

假设某一对桶中：

```text
a_1 有约 1000 万条 IP
b_1 有约 1000 万条 IP
```

每个 IP 原始占 4 字节：

```text
a_1：1000 万 × 4 字节 ≈ 40 MB
b_1：1000 万 × 4 字节 ≈ 40 MB
```

两个小文件的原始数据总量约为：

```text
40 MB + 40 MB = 80 MB
```

但这不表示算法必须同时把两个文件的 80 MB 全部装进内存。

实际处理时通常只把其中一个桶建立成哈希表，另一个桶逐条读取。

另外，哈希表会有额外开销，因此 40 MB 的原始 IP 数据放入 `unordered_set` 后，实际占用可能明显超过 40 MB。

所以实际设计时不能把桶大小卡在 100 MB 附近，必须预留足够余量。

---

## 六、桶会不会大小不均匀

会。

分桶后只能保证平均大小接近：

```text
总数据量 ÷ 桶数
```

不能保证每个桶完全一样大。

可能出现：

```text
0 号桶：较小
1 号桶：正常
2 号桶：较大
```

造成桶大小不均匀的原因包括：

- 哈希函数分布不好；
- 原始数据本身分布不均匀；
- 某些 IP 重复次数特别多；
- 桶数量太少。

解决方法：

### 1. 增加桶的数量

不要只按理论最小数量分桶，应多分一些桶，给内存留下余量。

例如可以分成：

```text
64、128、256 个桶
```

具体数量要根据数据大小和哈希表的实际内存开销决定。

### 2. 检查每个桶的文件大小

分桶后检查各个小文件。

如果某个桶仍然过大，就对这个桶再次使用另一种哈希规则进行二次分桶。

例如：

```text
a_7 太大
b_7 太大
```

再次拆分为：

```text
a_7_0、a_7_1、……、a_7_9
b_7_0、b_7_1、……、b_7_9
```

这种方法叫作递归分桶。

---

## 七、时间复杂度

假设文件 `a` 有 `n` 条数据，文件 `b` 有 `m` 条数据。

### 第一次分桶

两个文件都需要完整读取一遍：

```text
O(n + m)
```

### 第二次逐桶比较

所有桶中的数据仍然各处理一次：

```text
O(n + m)
```

因此整个算法的平均时间复杂度仍然是：

```text
O(n + m)
```

虽然单次哈希插入和查询的平均复杂度是 `O(1)`，但所有数据都必须至少被处理一次。

---

## 八、空间复杂度

内存中一次只保存一对桶中的一部分数据，因此空间复杂度为：

```text
O(最大桶中的数据量)
```

通过合理增加桶数，可以让最大桶的数据量控制在 100 MB 内。

需要注意，真正的限制应该以哈希表实际占用的内存为准，而不只是原始文件大小。

---

# 两道题目的区别

| 对比项 | 题目一：内存足够 | 题目二：内存限制 100 MB |
|---|---|---|
| 处理方式 | 整个文件建立哈希表 | 先分桶，再逐桶建立哈希表 |
| 是否需要磁盘小文件 | 不需要 | 需要 |
| 平均时间复杂度 | `O(n + m)` | `O(n + m)` |
| 空间复杂度 | `O(n)` | `O(最大桶大小)` |
| 核心思想 | 哈希查找 | 哈希分桶 + 哈希查找 |
| 主要问题 | 内存是否足够 | 桶是否过大、分布是否均匀 |

---

# 核心记忆

```text
内存足够：
把较小文件放入哈希表，扫描另一个文件。

内存不足：
先使用相同的哈希规则将两个文件分成多个对应的小桶，
再逐对比较同编号的桶。

同一个数据使用相同哈希函数和桶数，
一定会进入相同编号的桶。

单次哈希查询平均是 O(1)，
整个算法仍然是 O(n + m)。
```
