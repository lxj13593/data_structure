# 一致性哈希算法笔记

## 1. 一致性哈希解决什么问题

一致性哈希主要用于分布式系统中的：

- 数据分片
- 分布式缓存
- 负载均衡
- 分布式存储

它解决的核心问题是：

> 当服务器数量发生变化时，尽量减少需要重新分配的数据。

---

## 2. 普通取模哈希的问题

假设有 `n` 台服务器，普通做法是：

```cpp
index = hash(key) % n;
```

例如有 3 台服务器：

```text
hash(key) % 3
```

当增加一台服务器后：

```text
hash(key) % 4
```

因为取模的除数发生了变化，大量 key 的计算结果都会改变。

这会导致：

- 大量缓存失效
- 大量数据迁移
- 短时间内数据库压力增大
- 系统扩容和缩容成本较高

---

## 3. 一致性哈希的基本思想

一致性哈希不直接对服务器数量取模，而是把哈希值空间组织成一个环。

假设哈希值范围为：

```text
0 ～ 2³² - 1
```

首尾相连后形成哈希环：

```text
0 → 1 → 2 → ... → 2³²-1 → 0
```

然后将：

- 服务器节点
- 数据 key

都通过哈希函数映射到这个环上。

---

## 4. 服务器如何映射到环上

假设有三台服务器：

```text
hash(serverA) = 100
hash(serverB) = 400
hash(serverC) = 800
```

它们在环上的位置为：

```text
0 → A(100) → B(400) → C(800) → 回到0
```

---

## 5. 数据如何查找服务器

首先计算数据 key 的哈希值，然后从该位置开始，沿顺时针方向寻找第一台服务器。

例如：

```text
hash(key1) = 250
```

从位置 250 顺时针寻找，遇到的第一台服务器是：

```text
B(400)
```

所以：

```text
key1 → serverB
```

再例如：

```text
hash(key2) = 900
```

900 之后没有服务器，需要绕回环的起点，最终找到：

```text
A(100)
```

所以：

```text
key2 → serverA
```

核心规则：

> key 顺时针找到的第一台服务器，就是负责该 key 的服务器。

---

## 6. 每台服务器负责哪一段

假设环上的顺序为：

```text
A(100) → B(400) → C(800) → A(100)
```

那么：

```text
A负责：(800, 100]
B负责：(100, 400]
C负责：(400, 800]
```

可以理解为：

> 每个服务器负责它前一个节点之后，到自己所在位置之间的数据。

---

## 7. 增加服务器

原来有：

```text
A(100) → B(400) → C(800)
```

现在加入：

```text
D(300)
```

新的顺序为：

```text
A(100) → D(300) → B(400) → C(800)
```

加入 D 之前：

```text
(100, 400] 范围的数据属于 B
```

加入 D 之后：

```text
(100, 300] → D
(300, 400] → B
```

因此，只有原来属于 B 的一部分数据需要迁移到 D。

其他服务器负责的数据基本不受影响。

---

## 8. 删除服务器

假设删除：

```text
B(400)
```

原来 B 负责：

```text
(100, 400]
```

删除 B 后，这一段数据继续顺时针寻找，交给下一个服务器：

```text
C(800)
```

因此只有 B 上的数据需要迁移，其他节点的数据基本不用改变。

---

## 9. 普通哈希与一致性哈希对比

| 对比项 | 普通取模哈希 | 一致性哈希 |
|---|---|---|
| 分配方式 | `hash(key) % n` | 在哈希环上顺时针查找 |
| 服务器数量变化 | 大量 key 重新映射 | 只影响相邻区域 |
| 数据迁移量 | 较大 | 较小 |
| 扩缩容成本 | 较高 | 较低 |
| 适用场景 | 节点数量比较固定 | 分布式动态扩缩容 |

---

## 10. 一致性哈希存在的问题：数据倾斜

真实服务器经过哈希后，在环上的位置是随机的。

例如：

```text
A(100)
B(150)
C(900)
```

服务器负责的范围为：

```text
A负责：(900, 100]
B负责：(100, 150]
C负责：(150, 900]
```

可以看到：

- B 负责的范围很小
- C 负责的范围特别大

因此 C 可能承担大量数据，而 B 上的数据很少。

这种现象称为：

> 数据倾斜或负载不均衡。

---

## 11. 虚拟节点

为了减少数据倾斜，可以为一台真实服务器创建多个虚拟节点。

例如服务器 A 对应：

```text
A#1
A#2
A#3
```

服务器 B 对应：

```text
B#1
B#2
B#3
```

服务器 C 对应：

```text
C#1
C#2
C#3
```

这些虚拟节点分别经过哈希，分散到整个环上：

```text
A#1 → C#1 → B#1 → A#2 → B#2 → C#2 → ...
```

每个虚拟节点都指向对应的真实服务器：

```text
A#1 ┐
A#2 ├──→ serverA
A#3 ┘
```

数据查找到虚拟节点后，再存入对应的真实服务器。

例如：

```text
key → A#2 → serverA
```

---

## 12. 虚拟节点为什么能使数据更均匀

没有虚拟节点时，一台服务器只占据环上的一个位置，只负责一整块区域。

如果这个区域刚好很大，该服务器负载就很高。

使用虚拟节点后，一台服务器会负责多个分散的小区域：

```text
serverA负责：
一小段 + 一小段 + 一小段 + ...
```

虽然某一个小区域可能较大，但其他区域可能较小，多个区域相加后，整体更容易接近平均值。

可以理解为：

```text
一个真实节点 = 只随机一次
多个虚拟节点 = 随机很多次后求和
```

随机次数越多，偶然性越容易被平均掉。

因此虚拟节点的核心作用是：

> 将一个可能很大的连续区域，拆成多个分散的小区域，从而降低负载不均衡的概率。

虚拟节点越多，通常分布越均匀，但维护节点的开销也会增加。

---

## 13. 数据结构实现

哈希环上的节点需要按照哈希值有序保存。

可以使用：

```cpp
std::map<size_t, std::string>
```

其中：

```text
key   ：服务器或虚拟节点的哈希值
value ：对应的真实服务器
```

例如：

```cpp
std::map<size_t, std::string> hash_ring;
```

查找某个 key 对应服务器时，需要找：

> 第一个哈希值大于等于 `hash(key)` 的节点。

可以使用：

```cpp
auto it = hash_ring.lower_bound(hash_value);
```

如果找到了：

```cpp
return it->second;
```

如果没有找到，说明已经走到环的尾部，需要绕回起点：

```cpp
return hash_ring.begin()->second;
```

基本逻辑：

```cpp
auto it = hash_ring.lower_bound(hash_value);

if (it == hash_ring.end())
{
    it = hash_ring.begin();
}

return it->second;
```

---

## 14. 查找复杂度

假设哈希环上共有 `n` 个真实节点或虚拟节点。

使用红黑树结构的 `std::map`：

```text
查找：O(log n)
插入：O(log n)
删除：O(log n)
```

如果使用有序数组配合二分查找：

```text
查找：O(log n)
```

但是插入和删除节点通常需要移动元素。

---

## 15. 节点增加和删除的影响范围

### 增加节点

新节点只会从它顺时针方向的下一个节点那里接收一部分数据。

```text
新节点 ← 下一个节点的部分数据
```

### 删除节点

被删除节点的数据，会交给它顺时针方向的下一个节点。

```text
被删除节点的数据 → 下一个节点
```

因此一致性哈希中的变化通常只影响相邻节点。

---

## 16. 注意事项

### 1. 哈希函数需要分布均匀

如果哈希函数本身分布不均匀，即使使用虚拟节点，也可能出现数据倾斜。

### 2. 虚拟节点名字应不同

例如：

```text
serverA#0
serverA#1
serverA#2
```

分别计算哈希值，不能让所有虚拟节点使用完全相同的字符串。

### 3. 需要处理环形回绕

如果 `lower_bound()` 返回 `end()`，应该回到：

```cpp
hash_ring.begin()
```

### 4. 虚拟节点数量不是越多越好

虚拟节点增加会使分布更加均匀，但也会增加：

- 节点存储空间
- 插入和删除成本
- 环的维护成本

---

## 17. 核心总结

```text
一致性哈希
    = 哈希环
    + 顺时针查找
    + 虚拟节点
```

普通取模哈希：

```cpp
hash(key) % 服务器数量
```

服务器数量一旦改变，大量 key 的位置都会变化。

一致性哈希：

```text
服务器和数据都映射到哈希环
数据顺时针寻找第一台服务器
```

服务器增删时，只影响环上相邻的一小部分数据。

虚拟节点：

```text
一台真实服务器对应多个环上位置
```

作用是：

```text
将负载拆成多个分散的小区域
减少数据倾斜
使服务器负载更加均匀
```

最重要的三句话：

> key 顺时针遇到的第一台服务器负责该 key。  
> 增删节点时，只迁移相邻区域的数据。  
> 虚拟节点通过多个分散位置减少负载不均衡。
