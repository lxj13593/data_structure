# 位图（Bitmap）笔记

## 1. 位图是什么

位图是一种使用二进制位 `bit` 来记录数据是否存在的数据结构。

每一位只有两种状态：

```text
0：该数字没有出现
1：该数字出现过
```

位图特别适合处理：

- 非负整数；
- 数值范围明确；
- 数据量很大；
- 只需要判断“是否存在”；
- 去重、查重等问题。

---

## 2. 基本原理

假设需要记录数字 `0～7` 是否出现，只需要 1 个字节。

因为：

```text
1 Byte = 8 bit
```

例如：

```text
bit 下标：7 6 5 4 3 2 1 0
bit 值：  0 0 1 0 1 0 1 0
```

表示：

```text
数字 1、3、5 出现过
```

因为第 1、3、5 位为 `1`。

---

## 3. 一个数字对应哪个字节和哪一位

假设使用字节数组保存位图，一个字节有 8 位。

对于数字 `num`：

```text
字节下标 = num / 8
位下标   = num % 8
```

例如数字 `21`：

```text
21 / 8 = 2
21 % 8 = 5
```

说明：

```text
数字 21 保存在第 2 个字节的第 5 位
```

例如数字 `10`：

```text
10 / 8 = 1
10 % 8 = 2
```

说明：

```text
数字 10 保存在第 1 个字节的第 2 位
```

数组下标从 `0` 开始。

---

## 4. 插入、查找和删除

### 插入

插入数字时，把对应位设置为 `1`。

```text
0 → 1
```

### 查找

检查对应位是否为 `1`。

```text
是 1：数字存在
是 0：数字不存在
```

### 删除

把对应位重新设置为 `0`。

```text
1 → 0
```

这些操作通常都可以直接定位到目标位，因此时间复杂度是：

```text
O(1)
```

---

## 5. 位运算原理

假设数字 `num` 对应的位下标是：

```text
bit_index = num % 8
```

构造掩码：

```text
1 << bit_index
```

例如位下标为 `3`：

```text
00000001
左移 3 位
00001000
```

### 设置某一位为 1

使用按位或：

```text
原数据 | 掩码
```

### 判断某一位是否为 1

使用按位与：

```text
原数据 & 掩码
```

结果不为 `0`，说明该位是 `1`。

### 设置某一位为 0

先将掩码取反，再使用按位与：

```text
原数据 & (~掩码)
```

---

## 6. 为什么位图节省内存

普通 `int` 通常占：

```text
4 Byte = 32 bit
```

如果直接存储 1 亿个整数：

```text
1 亿 × 4 Byte = 4 亿 Byte
```

大约是：

```text
400 MB
```

如果只需要记录 `0～99999999` 是否出现，可以使用 1 亿位：

```text
1 亿 bit ÷ 8
= 1250 万 Byte
≈ 12.5 MB
```

所以位图记录存在状态时，比直接保存整数节省很多内存。

---

## 7. 2^32 bit 为什么等于 512 MiB

IPv4 地址可以用 32 位无符号整数表示，共有：

```text
2^32
```

种可能的地址。

如果为每个 IPv4 地址分配 1 个 bit，则需要：

```text
2^32 bit
```

因为：

```text
1 Byte = 8 bit = 2^3 bit
```

所以：

```text
2^32 bit ÷ 2^3
= 2^29 Byte
```

又因为：

```text
1 MiB = 2^20 Byte
```

因此：

```text
2^29 Byte ÷ 2^20
= 2^9 MiB
= 512 MiB
```

最终：

```text
2^32 bit = 512 MiB
```

按十进制单位计算约为：

```text
536.87 MB
```

---

## 8. 位图的优点

- 内存占用很小；
- 插入、删除、查询通常都是 `O(1)`；
- 非常适合大规模整数查重；
- 可以快速统计哪些数字出现过；
- 实现思路简单。

---

## 9. 位图的缺点

### 9.1 只能直接处理整数

字符串、浮点数、自定义对象不能直接作为位下标。

通常需要先转换为整数或哈希值。

### 9.2 数值范围过大会浪费空间

例如只存两个数字：

```text
1
10 亿
```

为了表示数字 `10 亿`，仍然需要开辟至少 10 亿位。

所以位图更适合：

```text
数据数量较多，而且数值范围相对集中
```

### 9.3 普通位图不能记录出现次数

普通位图只有：

```text
0：没出现
1：出现过
```

无法区分：

```text
出现 1 次
出现 2 次
出现 100 次
```

如果需要记录次数，可以使用：

- 多个位图；
- 计数位图；
- 数组；
- `unordered_map`。

### 9.4 普通位图不能保留重复元素

同一个数字插入多次，对应位仍然只是 `1`。

---

## 10. 位图和哈希表的区别

| 对比项 | 位图 | 哈希表 |
|---|---|---|
| 适合的数据 | 有明确范围的整数 | 整数、字符串、自定义类型 |
| 查询复杂度 | `O(1)` | 平均 `O(1)` |
| 内存占用 | 很小 | 较大 |
| 是否保存原始数据 | 通常不直接保存 | 保存键或键值对 |
| 是否能记录次数 | 普通位图不能 | 可以 |
| 是否适合稀疏大范围数据 | 不太适合 | 更适合 |
| 是否支持字符串 | 不直接支持 | 支持 |

---

## 11. 位图适合解决的问题

### 判断某个整数是否出现过

```text
给定大量整数，快速判断某个整数是否存在。
```

### 大量整数去重

将每个出现过的数字对应位置设为 `1`，最后遍历所有位。

### 查找缺失数字

将出现过的数字标记为 `1`，状态仍为 `0` 的位置就是缺失数字。

### 判断是否存在重复数字

插入一个数字前检查对应位：

```text
如果是 0：第一次出现
如果是 1：说明重复
```

### 海量数据查重

当数据是有限范围内的整数，并且内存有限时，位图通常比哈希表更节省内存。

---

## 12. 位图与 IPv4

IPv4 可以转换为一个 32 位无符号整数，因此理论上可以直接用 IPv4 的整数值作为位图下标。

IPv4 的完整范围有：

```text
2^32
```

个可能值。

完整位图需要：

```text
512 MiB
```

如果内存限制只有 100 MB，不能直接建立完整 IPv4 位图。

可以考虑：

- 哈希分桶后对每个桶使用位图；
- 使用压缩位图；
- 使用外部排序；
- 使用其他更节省空间的数据结构。

---

## 13. 核心记忆

```text
位图使用 1 个 bit 表示一个整数是否存在。

0 表示不存在，1 表示存在。

字节下标 = num / 8
位下标   = num % 8

插入、查询、删除通常都是 O(1)。

位图非常省内存，但适合数值范围明确、相对集中的非负整数。

普通位图只能表示“出现或未出现”，不能直接记录出现次数。
```
